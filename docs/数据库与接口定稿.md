# 数据库与接口定稿

本文件是开发期间的数据库和 API 唯一口径。实现代码时优先参考 `sql/init_city_repair.sql`。

## 数据库定稿

数据库：MySQL 8

核心业务表：

| 表名 | 说明 | 主责 |
|---|---|---|
| `sys_user` | 用户基础信息 | 组长 |
| `sys_role` | 角色字典 | 组长 |
| `sys_user_role` | 用户角色关系 | 组长 |
| `repair_category` | 报修类别 | 孔祥洋 |
| `repair_order` | 工单主表 | 全员共用，组长审核变更 |
| `order_assignment` | 分派记录 | 闫龙飞、袁翊博 |
| `order_status_log` | 状态日志 | 全员共用 |
| `order_attachment` | 报修前/处理后图片 | 苟佳豪、袁翊博 |
| `order_evaluation` | 居民评价 | 张天童 |
| `user_notification` | 用户通知 | 张天童 |

任何成员不得私自改表名、核心字段名、枚举值。确实要改时，先让组长更新 `sql/init_city_repair.sql` 和本文档。

## 状态流转

```text
PENDING_REVIEW -> PENDING_ASSIGN
PENDING_REVIEW -> REJECTED
PENDING_REVIEW -> CANCELLED
PENDING_ASSIGN -> PENDING_ACCEPT
PENDING_ACCEPT -> PROCESSING
PROCESSING -> COMPLETED
COMPLETED -> EVALUATED
```

终态：`EVALUATED`、`REJECTED`、`CANCELLED`

禁止新增同义状态。

## 基础接口

### 登录与当前用户

```http
POST /api/auth/login
GET  /api/auth/me
POST /api/auth/logout
```

登录后所有业务接口从登录上下文取当前用户，不允许信任前端传入的 `userId`。

### 类别

```http
GET    /api/categories
POST   /api/categories
PUT    /api/categories/{id}
PUT    /api/categories/{id}/status
DELETE /api/categories/{id}
```

已被工单引用的类别不能物理删除，应返回错误并引导停用。

## 居民报修接口

主责：苟佳豪

```http
POST /api/repair-orders
GET  /api/repair-orders/my
GET  /api/repair-orders/{id}
PUT  /api/repair-orders/{id}/cancel
POST /api/repair-orders/{id}/attachments
```

创建工单请求：

```json
{
  "categoryId": 1,
  "title": "小区门口道路坑洼",
  "location": "幸福小区南门",
  "description": "雨天积水影响通行",
  "contactPhone": "13800000002"
}
```

后端固定：

- `resident_id` 从登录上下文取；
- 初始状态为 `PENDING_REVIEW`；
- 初始优先级为 `NORMAL`，管理员后续可调整；
- 创建后写 `order_status_log`。

取消规则：

- 只能取消本人订单；
- 只能取消 `PENDING_REVIEW`；
- 成功后状态为 `CANCELLED` 并写日志。

## 管理员审核与分派接口

主责：闫龙飞

```http
GET /api/admin/orders
PUT /api/admin/orders/{id}/approve
PUT /api/admin/orders/{id}/reject
PUT /api/admin/orders/{id}/assign
GET /api/repair-orders/{id}/logs
```

审核通过请求：

```json
{
  "priority": "HIGH",
  "remark": "影响通行，优先处理"
}
```

驳回请求：

```json
{
  "reason": "位置信息不完整，请补充后重新提交"
}
```

分派请求：

```json
{
  "workerId": 3,
  "remark": "请今天内处理"
}
```

必须校验：

- 当前用户是 `ADMIN`；
- 只有 `PENDING_REVIEW` 可审核；
- 驳回必须有原因；
- 只有 `PENDING_ASSIGN` 可分派；
- `workerId` 必须存在且拥有 `WORKER` 角色；
- 分派时同一事务写 `repair_order`、`order_assignment`、`order_status_log`。

## 维修人员处理接口

主责：袁翊博

```http
GET  /api/worker/orders
PUT  /api/worker/orders/{id}/accept
PUT  /api/worker/orders/{id}/start
POST /api/worker/orders/{id}/progress
PUT  /api/worker/orders/{id}/complete
POST /api/worker/orders/{id}/attachments
```

进度请求：

```json
{
  "content": "已到现场，确认需要更换井盖"
}
```

完成请求：

```json
{
  "completionResult": "已完成井盖更换，道路恢复正常"
}
```

必须校验：

- 当前用户是 `WORKER`；
- 当前用户是工单当前有效分派人；
- `PENDING_ACCEPT` 才能接单；
- 接单或开始处理后进入 `PROCESSING`；
- `PROCESSING` 才能完成；
- 完成必须有处理说明；
- 状态更新和日志写入在同一事务。

## 评价与通知接口

主责：张天童

```http
POST /api/repair-orders/{id}/evaluation
GET  /api/evaluations/my
GET  /api/admin/evaluations
GET  /api/notifications/my
PUT  /api/notifications/{id}/read
```

评价请求：

```json
{
  "rating": 5,
  "content": "处理及时，结果满意"
}
```

必须校验：

- 当前用户是该工单居民；
- 工单状态必须是 `COMPLETED`；
- 评分范围是 1 到 5；
- 同一工单只能评价一次；
- 创建评价和工单进入 `EVALUATED` 在同一事务。

通知至少覆盖：

- 审核通过或驳回；
- 工单分派；
- 维修人员接单；
- 工单完成。

通知只能由接收人查看。

## 统计接口

主责：孔祥洋

```http
GET /api/statistics/overview
GET /api/statistics/status
GET /api/statistics/category
GET /api/statistics/trend
GET /api/statistics/worker-rank
```

统计口径：

- 工单总数：`repair_order` 总记录数；
- 待处理数量：`PENDING_REVIEW`、`PENDING_ASSIGN`、`PENDING_ACCEPT`、`PROCESSING`；
- 已完成数量：`COMPLETED`、`EVALUATED`；
- 完成率：已完成数量 / 工单总数；
- 最近 7 天趋势：按 `repair_order.created_at` 统计；
- 维修人员排行：按 `repair_order.current_worker_id` 和完成状态统计。

所有图表必须来自后端 API 和真实 SQL，禁止前端写死统计值。

## 必测异常

- 居民访问他人工单；
- 居民取消非 `PENDING_REVIEW` 工单；
- 非管理员调用审核接口；
- 重复审核；
- 分派给非维修人员；
- 维修人员处理他人工单；
- 重复完成工单；
- 未完成工单评价；
- 重复评价；
- 删除已被引用类别。
